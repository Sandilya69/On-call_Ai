1. Design Philosophy
OnCall Maestro's frontend is built for engineers who use it under pressure — at 3am, during production incidents, half-asleep. Every design decision serves one goal: get the engineer the information they need and the action they need to take in under 3 seconds. Nothing decorative. Nothing in the way.
1.1 Design Inspiration — Multi-Source Approach
The design borrows specific patterns from multiple best-in-class developer tools. No single source is copied — specific elements from each are combined into a system that feels familiar but is distinctly Maestro.
Inspiration Source
What We Borrow
What We Change
Linear (linear.app)
Keyboard-first navigation, ultra-dense data tables, zero padding on list items, instant search
Linear uses purple brand — we use your color. Linear is project mgmt — we adapt density for alert triage
Vercel Dashboard
Status indicators, deployment timeline rows, monospace metadata, clean header with breadcrumbs
Vercel is light mode first — we are dark mode first. Their cards are too spaced for our density needs
Raycast (raycast.com)
Command palette design, keyboard shortcut badges, result list hover states, compact icon+text rows
Raycast is a launcher — we adapt their list pattern for incident lists and search
PlanetScale Dashboard
Database row styling, connection string masked reveal, status badge system, metric cards
PlanetScale's nav is too heavy — we strip it down. We take their badge language for severity tags
Resend (resend.com)
Log viewer pattern, monospace event timeline, minimal sidebar, clean empty states
Resend's color palette is too neutral — we need severity colors to communicate urgency instantly
Clerk Dashboard
Team member rows, permission badges, avatar + name + role pattern, settings page layout
Clerk's tables are spaced for non-technical users — we tighten everything for SRE density
1.2 Design Principles — Non-Negotiable
Density over breathing room: Engineers need to see 20 incidents at a glance, not 5. Default table rows are 36px tall, not 56px. No wasted whitespace between data rows.
Status is always visible: Severity badges (P1/P2/P3/P4) and status indicators (FIRING/ACK/RESOLVED) are the most important elements on any page. They are always the highest contrast element in their row.
Keyboard first: Every action has a keyboard shortcut. Power users never touch the mouse. K to open command palette, J/K to navigate lists, A to acknowledge, R to resolve, E to escalate.
Monospace for data: Timestamps, IDs, API keys, fingerprints, durations — all monospace. Regular text for labels and descriptions. This visual distinction makes scanning fast.
No decorative elements: No gradients on cards. No shadows on everything. No rounded-2xl on buttons. Flat, sharp, functional. Rounded corners only where they aid hierarchy.
Dark mode is the default: Engineers look at dashboards in dark environments. Dark mode is not a toggle — it is the designed state. Light mode can be added later but is not v1.
Empty states are helpful: When a table is empty it does not just say 'No results'. It explains why and what to do next. Every empty state has an action.

Important Note on Colors
This document intentionally leaves all brand colors blank. The CSS variable names and their roles are defined below. You fill in the actual hex values. Every component references these variables — change the variable, the entire app updates.

5.2 Sidebar Design
Inspired by Linear's sidebar — compact, no icons-only mode in v1, always shows label. Active item has a left border accent, not a background fill. This avoids the 'pill nav' look that every generic dashboard uses.
Sidebar structure:
  [Logo + wordmark]
  ─────────────────
  ⬡  Dashboard
  ▣  Incidents       [badge: count of active P1/P2]
  ◫  Rota
  ◉  Engineers

Hover state:   bg subtle only, no border

5.3 Header Design
Inspired by Vercel's dashboard header — very minimal. Left side shows breadcrumb. Center is empty or has page search. Right has command palette trigger and user avatar.
Header structure (48px height, border-bottom: 1px border-default):
  [Team name] / [Current page]      [⌘K]  [🔔]  [Avatar]
  ←── Breadcrumb (14px mono) ──→           ←── Actions ──→
⌘K button: opens command palette (see Section 9)
🔔 bell: shows count of unread escalation notifications
Avatar: dropdown with profile, settings, sign out

// hooks/useRealtime.ts
import { useEffect } from 'react';
import { supabase }  from '@/lib/supabase';
import { useQueryClient } from '@tanstack/react-query';
export function useIncidentRealtime(teamId: string) {
  const queryClient = useQueryClient();
  useEffect(() => {
    const channel = supabase
      .channel(`incidents-${teamId}`)
      .on('postgres_changes', {
        event:  '*',
        schema: 'public',
        table:  'Incident',
        filter: `teamId=eq.${teamId}`,
      }, (payload) => {
        // Invalidate and refetch incidents query
        queryClient.invalidateQueries({ queryKey: ['incidents', teamId] });
      })
      .subscribe();
    return () => { supabase.removeChannel(channel); };
  }, [teamId, queryClient]);
}

1. Mobile number: Twilio calls this for P1 voice escalation. Without it voice calls fail.
2. Skill tags: Claude uses these to route the right alert to the right engineer. Without them Maestro is PagerDuty.
3. Timezone: Rota engine and fairness scoring depend on this being accurate.

Make sure the onboarding UI explains WHY each field matters. Engineers who understand the reason will fill it in properly.
