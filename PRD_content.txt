
OnCall Maestro
Product Requirements Document
AI-Powered On-Call Scheduler, Escalation Bot & Voice Handover Platform

This document defines the complete product, technical architecture, database schema, automation workflows, and build roadmap for OnCall Maestro — a direct competitor to PagerDuty, OpsGenie, and VictorOps.

1. Executive Summary
OnCall Maestro is an AI-native, self-hosted on-call management platform that replaces legacy tools like PagerDuty and OpsGenie. It combines intelligent alert routing, automated multi-channel escalation, AI-generated voice handovers, and smart rota scheduling into a single cohesive system — at 90% lower cost than existing SaaS incumbents.

1.1  Problem Statement
Engineering teams running 24/7 on-call rotations suffer from four critical failures in existing tools:
Alert fatigue — PagerDuty fires raw alerts without intelligence. Engineers receive 800+ alerts per week, 90%+ of which are noise.
Dumb routing — Round-robin assignment ignores skills, timezones, and incident history. The wrong person gets woken up.
Zero handover context — Shift changes happen with no structured briefing. Incoming engineers start blind.
Vendor lock-in at high cost — $40/user/month, all data trapped, no AI, no export.

1.2  Solution Overview
OnCall Maestro addresses every failure above with a modern, AI-first architecture:
Intelligent deduplication and grace-period filtering eliminates noise before any notification fires.
Claude AI routes each alert to the best available engineer based on skill tags, timezone, and incident history — not just rota position.
Celery-powered escalation timers enforce SLA windows automatically. No manual policy management.
AI-generated voice handover briefings via Whisper TTS ensure incoming engineers have full context on every shift change.
Full self-hosting with Supabase (PostgreSQL) means teams own all data with no per-seat pricing.

2. Product Scope & Features
2.1  Core Modules
Module 1: Alert Ingestion Engine
The entry point for all signals. Every alert from any monitoring tool hits a single webhook endpoint. Maestro normalizes, deduplicates, and fingerprints before any downstream logic runs.
Universal webhook receiver — accepts Prometheus AlertManager, PagerDuty, Grafana, custom HTTP POST
Alert normalization — maps any schema to a canonical AlertObject (id, title, severity, service, labels, timestamp)
Fingerprint deduplication — SHA-256 hash of (service + metric + labels) checked against Redis with 30-second TTL
Grace period filter — P3/P4 alerts held for 45 seconds; if self-resolved, no notification fires
Severity re-evaluation — Claude API optionally re-classifies severity based on recent incident history for the same service

Module 2: AI Routing Engine
The intelligence layer. Claude receives the normalized alert and selects the optimal engineer from the on-call pool using a structured reasoning prompt.
Input context: alert title, service, severity, error type, current rota, each engineer's skill tags, timezone, and last 5 incidents they resolved
Output: primary assignee ID, backup chain (ordered list), estimated complexity score, routing rationale (logged for audit)
Fallback: if Claude API is unavailable (>2s timeout), system falls back to standard rota order — never blocks
Cost: approximately $0.008 per routing decision using claude-sonnet-4-6

Module 3: Multi-Channel Notification
Notifications fire in parallel across channels. Channel priority is configurable per team and per severity level.

Module 4: Escalation Timer Engine
Every unacknowledged alert has a live countdown timer managed by Celery Beat. Escalation is fully automatic with zero manual intervention required.
Timer starts the moment the first notification fires
ACK can be received from: Slack reaction, /ack slash command, web dashboard click, or API call
Cross-channel ACK state stored in Redis — ACK from any channel stops all escalation timers
Before escalating, system checks: assignee calendar for OOO/travel blocks, current shift status, escalation policy exceptions
Escalation chain: Primary → L2 backup → Team Lead → Incident Commander → Manager
Every escalation event is logged with reason, timestamp, and which engineer was tried

Module 5: Voice Handover Briefing
10 minutes before every shift end, Maestro auto-generates a spoken briefing for the incoming engineer. No manual runbooks. No rushed Slack summaries.
Claude generates a structured script: open incidents (status + last action), recently resolved (with what fixed it), watch items (services showing elevated error rates), and any unusual patterns from the shift
Whisper TTS converts the script to a natural-sounding MP3 audio file
Delivered via: Slack audio message to incoming engineer, optional phone call, stored in handover history
Incoming engineer can reply with /briefing to regenerate or /extend to request a written summary

Module 6: Rota Management & Scheduling
Smart scheduling with conflict detection, timezone fairness scoring, and Google Calendar synchronisation.
Engineers set availability windows via Slack (/available) or web dashboard
Constraint solver generates weekly rota respecting: availability, timezone balance (no one always on overnight), holiday calendar, maximum consecutive shifts (configurable)
/swap @engineer command in Slack — Maestro checks both calendars, approves or flags conflicts, updates rota and Calendar events
Manager approval workflow for swaps exceeding a configurable threshold (e.g. more than 2 swaps per week)
Google Calendar sync — every shift creates a Calendar event with on-call runbook links in the description

4. Database Schema (PostgreSQL)
All tables use UUID primary keys, created_at/updated_at timestamps, and soft deletes (deleted_at). Row Level Security (RLS) is enabled on all tables in Supabase to enforce team-level isolation.

4.1  organisations
Top-level multi-tenancy entity. All other tables belong to an organisation.

4.2  engineers
Individual users. An engineer belongs to one organisation and can be a member of multiple teams.

4.3  teams

4.4  team_members
Junction table linking engineers to teams with their role.

4.5  escalation_policies
Defines the timing and chain for auto-escalation. One policy can apply to multiple teams.

4.6  rota
Each row is a single on-call shift assignment. The rota is generated weekly by the scheduling engine.

4.7  incidents
The central record of every alert that entered the system. All downstream events reference this table.

4.8  escalation_events
Immutable audit log of every escalation action. Never updated, only appended.

4.9  notifications
Log of every outbound notification across all channels.

4.10  handovers
Record of every AI-generated shift handover briefing.

4.11  availability
Engineers declare when they are NOT available for on-call shifts.

4.12  audit_log
Append-only log of every user and system action across the platform. Used for compliance, debugging, and security review.

4.13  Key Indexes
Performance-critical indexes beyond primary keys:
incidents(org_id, status, created_at DESC) — dashboard incident list query
incidents(fingerprint) — dedup lookup on alert ingest
incidents(service, created_at DESC) — service history for AI routing context
rota(team_id, start_time, end_time) — current on-call lookup
escalation_events(incident_id, created_at DESC) — escalation timeline
audit_log(org_id, created_at DESC) — compliance queries

5. Automation Workflows
5.1  Alert Ingest → Route → Notify (Primary Flow)
This is the hottest path in the system. Every step is async and non-blocking. Target total latency from webhook receipt to Slack notification: under 3 seconds for P1.

Webhook POST arrives at POST /api/v1/ingest/{org_slug}
FastAPI parses and validates payload against AlertSchema (Pydantic). Returns 202 Accepted immediately — never blocks the sender.
Celery task process_alert(payload, org_id) is queued in Redis.
Dedup check: SHA-256 fingerprint(service + metric + labels). SETNX in Redis with 30-second TTL. If key exists, task exits — duplicate suppressed.
Severity grace period: if P3 or P4, sleep task for 45 seconds (Celery countdown). If alert is resolved before wakeup, no notification fires.
Persist incident record to PostgreSQL with status=open.
AI routing: call Claude API with routing prompt (alert context + engineer pool). Parse response for assignee_id. On timeout (>2s), fall back to rota order.
Update incident.assignee_id, log routing_rationale.
Dispatch notifications in parallel: Celery group([send_slack.s(), send_sms.s(), send_voice.s()]) — channels are filtered by severity config.
Start escalation countdown: escalation_timer.apply_async(args=[incident_id], countdown=ack_window_seconds)
Write escalation_events row (level=1, channel used, delivery_status).

5.2  Escalation Timer Workflow
Runs as a Celery task. Designed to be idempotent — safe to retry on failure.

Task wakes up after ACK window expires.
Query incident.status. If acknowledged or resolved, exit immediately — no action.
Check assignee's Google Calendar for OOO block overlapping current time. If OOO found, skip to next engineer in chain.
Determine next escalation target from escalation_policy.escalation_chain.
If max_escalations reached, notify manager directly and post to team Slack channel with @here.
Update incident.assignee_id to new engineer.
Dispatch notifications to new assignee (same parallel channel dispatch as primary flow).
Log escalation_events row (reason=no_ack, level=N+1).
Start new escalation countdown for the new assignee.

5.3  ACK Reception Workflow
ACK can arrive from multiple channels simultaneously. System is idempotent — double-ACK has no side effects.

ACK signal received (Slack /ack {incident_id}, Slack reaction, API PATCH /incidents/{id}/ack, web dashboard button).
Redis SET ack:{incident_id} = engineer_id, EX 86400 (1 day TTL for audit).
Celery task: cancel active escalation timer for this incident (using revoke by task_id stored at schedule time).
UPDATE incidents SET status=acknowledged, acknowledged_at=NOW(), assignee_id={engineer_id}.
Append audit_log row (action=incident.acknowledged).
Post confirmation to team Slack channel: incident title, acknowledged by, time to ACK.

5.4  Voice Handover Briefing Workflow
Runs on a Celery Beat schedule — triggered 10 minutes before every shift end time.

Celery Beat runs check_upcoming_handovers every 5 minutes.
Query rota for shifts ending in the next 10–15 minutes. For each: fetch outgoing engineer, incoming engineer.
Query open incidents for the team. Query incidents resolved this shift. Identify services with elevated error rate in the last 2 hours.
Build Claude prompt with structured data: open_incidents[], resolved_today[], watch_services[], shift_duration, team_name.
Call Claude API. Parse response into structured briefing sections: Overview, Open Incidents, Resolved Today, Watch List, Handover Notes.
Call Whisper TTS API with briefing text. Receive MP3 audio. Upload to object storage. Store URL in handovers.audio_url.
Send Slack audio message to incoming engineer's DM with briefing text as fallback caption.
Optionally: initiate Twilio voice call to incoming engineer's phone if they have voice_handover=true in notification_prefs.
Insert handovers row with both engineer IDs, briefing text, audio URL, delivery status.

5.5  Rota Generation Workflow
Runs every Sunday at 18:00 UTC for the following week. Also runs on-demand when /swap or availability changes occur.

Fetch all engineers in the team with is_active=true.
Fetch availability table for the target week — exclude engineers with unavailable_from/to overlapping any shift slot.
Apply constraint rules: max_shifts_per_week per engineer, no two consecutive night shifts, timezone balance scoring (engineers in UTC-5 to UTC+2 get preference for night shifts).
Run constraint solver (simple greedy algorithm with backtracking). Output: list of {shift_slot → engineer_id} assignments.
Write rota rows to database for each assignment.
For each rota row, call Google Calendar API to create calendar event on engineer's calendar with shift details and runbook links.
Post weekly rota summary to team Slack channel.
Write audit_log row (action=rota.generated, metadata={week, team_id, assignments_count}).

5.6  Swap Request Workflow
Triggered by /swap @engineer in Slack.

Slack Bolt receives slash command. Parse: requesting engineer, target engineer, optional shift date.
Find the requesting engineer's upcoming shift.
Check target engineer's availability table for that shift window — conflict check.
Check both engineers' max_shifts_per_week would not be exceeded.
If auto-approve criteria met (no conflicts, both under shift cap, more than 24h notice): approve immediately.
If manual approval required: send Slack message to team lead with Approve / Reject buttons.
On approval: UPDATE rota SET engineer_id = target_engineer_id, generated_by='swap', notes='Swapped with {name}'. Update Google Calendar events for both engineers.
Notify both engineers of confirmed swap via Slack DM.
Log audit_log row (action=rota.swap_approved).

7. AI Prompt Specifications
7.1  Alert Routing Prompt
Sent to claude-sonnet-4-6 for every new unrouted incident. Must return structured JSON.

System: You are an expert on-call routing system. Your job is to select the best engineer to handle an incident.
Always respond with valid JSON only. No preamble.

User: Route this incident to the best available engineer.

INCIDENT:
Title: {incident.title}
Severity: {incident.severity}
Service: {incident.service}
Description: {incident.description}
Labels: {incident.labels}

AVAILABLE ENGINEERS:
{engineers.map(e => JSON.stringify({
  id: e.id, name: e.name, skills: e.skill_tags,
  timezone: e.timezone, is_working_hours: <bool>,
  recent_incidents_resolved: <count_last_7_days>
}))}

RECENT INCIDENTS ON THIS SERVICE (last 5):
{recent_incidents}

Respond with:
{
  "assignee_id": "<uuid>",
  "backup_chain": ["<uuid>", "<uuid>"],
  "confidence": 0.0-1.0,
  "rationale": "<1-2 sentence explanation>"
}

7.2  Voice Handover Briefing Prompt
Sent to claude-sonnet-4-6 at shift end. Output is spoken aloud by Whisper TTS — must be written as natural speech, not bullet points.

System: You are generating a spoken handover briefing for an on-call engineer.
Write it as natural speech — clear, concise, and actionable. No bullet points.
Target length: 45-75 seconds when read aloud (approximately 120-200 words).

User: Generate a shift handover briefing.

OUTGOING: {outgoing_engineer.name} (shift: {shift_start} to {shift_end})
INCOMING: {incoming_engineer.name}
TEAM: {team.name}

OPEN INCIDENTS ({open_incidents.length}):
{open_incidents.map(i => `[{i.severity}] {i.title} - open for {i.duration_minutes}min, 
last action: {i.last_action}`)}

RESOLVED THIS SHIFT ({resolved.length}):
{resolved.map(i => `{i.title} - fixed by {i.resolution_notes}`)}

WATCH LIST (services with elevated errors):
{watch_services}

Write the briefing as if you are {outgoing_engineer.name} speaking directly to {incoming_engineer.name}.

10. Security & Compliance
10.1  Security Requirements
All API traffic over HTTPS (TLS 1.2 minimum). No HTTP in production.
Webhook signatures verified via HMAC-SHA256 on every ingest request.
JWT tokens signed RS256, expire after 8 hours. Refresh tokens stored httpOnly.
Engineer phone numbers and calendar OAuth tokens encrypted at rest (AES-256 via Supabase Vault).
Row Level Security enforced on all Supabase tables — verified by automated test suite.
All secrets in environment variables. No credentials in source code or Docker images.
Dependency scanning via GitHub Actions (pip audit, npm audit) on every PR.

10.2  GDPR Compliance
Engineer PII (name, email, phone) is exportable via GET /api/v1/engineers/{id}/export.
Engineer deletion (right to erasure) anonymizes records rather than hard-deleting — incident history is preserved with engineer_id nulled.
Data retention policy: incidents and logs retained 12 months hot, 5 years in cold storage, then purged.
No third-party analytics or tracking pixels in the web dashboard.

10.3  Operational Security
Principle of least privilege: Celery workers have read/write access only to tables they need.
Audit log is append-only — no UPDATE or DELETE permitted on audit_log table (enforced by Postgres policy).
Rate limiting on all public endpoints: 100 req/min per IP, 1000 req/min per org.
Slack slash commands verified using Slack signing secret on every request.

Escalation | Automatic reassignment of an incident to a higher-level engineer after the ACK window expires.

Rota | The on-call schedule — which engineer is responsible during which time window.
